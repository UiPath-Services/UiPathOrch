# Get-OrchTestCase

## SYNOPSIS
Gets test case definitions from the Orchestrator.

## SYNTAX

```
Get-OrchTestCase [[-Name] <String[]>] [-Path <String[]>] [-Recurse] [-Depth <UInt32>]
 [-ProgressAction <ActionPreference>] [<CommonParameters>]
```

## DESCRIPTION
The Get-OrchTestCase cmdlet retrieves test case definitions from the Orchestrator environment. Test cases are workflow files (.xaml) that have been designated as test cases within automation packages and can be executed as part of test sets for automated testing scenarios.

This cmdlet operates on folder entities and supports recursive traversal to find test cases across different folder hierarchies. Test cases are associated with specific packages and versions, enabling version-controlled testing and quality assurance processes.

Test case definitions include metadata about the associated package, version information, creation and modification details, and feed associations. These test cases can be organized into test sets and executed to validate automation workflows.

Primary Endpoint: GET /odata/TestCaseDefinitions

OAuth required scopes: OR.TestSets or OR.TestSets.Read

Required permissions: TestSets.View

## EXAMPLES

### Example 1
```powershell
PS Orch1:\Shared> Get-OrchTestCase
```

Retrieves all test case definitions from the current folder.

### Example 2
```powershell
PS Orch1:\Shared> Get-OrchTestCase *Test*.xaml
```

Retrieves all test case definitions whose names contain "Test" using wildcard matching.

### Example 3
```powershell
PS Orch1:\> Get-OrchTestCase -Recurse
```

Retrieves all test case definitions from the current folder and all subfolders recursively.

### Example 4
```powershell
PS C:\> Get-OrchTestCase -Path Orch1:\Production -Recurse | ConvertTo-Json -Depth 2
```

Retrieves test case definitions from the Production folder and displays detailed structure in JSON format.

## NOTES

This cmdlet operates on folder entities, which means you may need to navigate to specific folders or use the -Path parameter to target the desired location.

When using the -Recurse parameter with large folder hierarchies, the operation may take considerable time to complete. Consider using -Depth to limit the traversal scope if needed.

Test case definitions are associated with automation packages and contain important version information:
- PackageIdentifier: The automation package containing the test case
- AppVersion: The application version of the test case
- CreatedVersion: The package version when the test case was created
- LatestVersion: The most recent version of the associated package
- FeedId: The package feed association

Parameter placement is important for folder entity cmdlets. Place -Path, -Recurse, -Depth parameters immediately after the cmdlet name for proper auto-completion functionality.

Use ConvertTo-Json to explore the complete structure of test case definition objects, as they contain detailed metadata including version tracking and package associations.

Test case definitions serve as the foundation for automated testing workflows and can be organized into test sets for comprehensive quality assurance processes.
