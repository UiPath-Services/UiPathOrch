# Enable-OrchUserAttended

## SYNOPSIS
Enables attended automation for specified users.

## SYNTAX

```
Enable-OrchUserAttended [-UserName] <String[]> [-Path <String[]>] [-ProgressAction <ActionPreference>]
 [-WhatIf] [-Confirm] [<CommonParameters>]
```

## DESCRIPTION
The Enable-OrchUserAttended cmdlet enables attended automation capabilities for specified users in the Orchestrator environment. Attended automation allows users to work collaboratively with robots, executing processes in their user sessions while maintaining user interaction and supervision.

When enabled, users can run robots in attended mode, which means the robots execute in the user's active session and can interact with applications and interfaces that the user has access to. This is particularly useful for scenarios where human oversight or interaction is required during process execution.

The cmdlet requires the UserName parameter to specify which users should have attended automation enabled. Multiple users can be specified simultaneously. The operation affects user permissions and capabilities within the Orchestrator tenant.

This setting is essential for organizations implementing attended automation scenarios where users and robots collaborate to complete business processes efficiently.

Primary Endpoint: POST /odata/ProcessSchedules/UiPath.Server.Configuration.OData.SetEnabled

OAuth required scopes: OR.Jobs

Required permissions: Schedules.Edit

## EXAMPLES

### Example 1
```powershell
PS C:\> Enable-OrchUserAttended john.doe
```

Enables attended automation for the user "john.doe". Uses positional parameter for the username.

### Example 2
```powershell
PS C:\> Enable-OrchUserAttended -UserName alice.smith, bob.jones
```

Enables attended automation for multiple users simultaneously. This allows both alice.smith and bob.jones to run robots in attended mode.

### Example 3
```powershell
PS C:\> Enable-OrchUserAttended -UserName marketing.team -Path Orch1:, Orch2:
```

Enables attended automation for the marketing.team user across multiple Orchestrator instances. Useful in multi-tenant environments.

### Example 4
```powershell
PS C:\> Enable-OrchUserAttended -UserName admin.user -WhatIf
```

Shows what would happen if attended automation was enabled for admin.user without actually making the change. Useful for verification before execution.

### Example 5
```powershell
PS C:\> Get-OrchUser -UserName *automation* | Enable-OrchUserAttended -Confirm
```

Finds all users with "automation" in their username and enables attended automation for them with confirmation prompts for each user.

## NOTES

Attended automation must be properly licensed and configured in your Orchestrator environment before users can utilize this capability.

This cmdlet modifies user permissions and capabilities. Ensure that users have the appropriate licenses and that your organization's security policies allow attended automation for the specified users.

Attended automation enables users to run robots in their active sessions, which means the robots will have the same access permissions as the user. Consider security implications when enabling this feature.

The opposite operation can be performed using Disable-OrchUserAttended to revoke attended automation capabilities from users.

Use -WhatIf to preview changes before applying them, especially when enabling attended automation for multiple users.
