# Get-TmTestCase

## SYNOPSIS
Gets test cases from projects in Test Manager.

## SYNTAX

```
Get-TmTestCase [[-Name] <String[]>] [-Path <String[]>] [-Recurse] [-ProgressAction <ActionPreference>]
 [<CommonParameters>]
```

## DESCRIPTION
The Get-TmTestCase cmdlet retrieves test cases from Test Manager projects. Test cases are individual test scenarios that can be either manual or automated, defining specific test steps, expected results, and automation associations.

This cmdlet operates on folder entities and requires navigation to specific Test Manager projects or the use of -Path, -Recurse parameters. It works with the UiPathOrchTm provider, which is automatically available when Test Manager scopes are included in the configuration.

Test cases can be associated with automation projects and contain information about test execution parameters, preconditions, and version history. They serve as the foundation for test execution within test sets and provide detailed test documentation and traceability.

Primary Endpoint: GET /testmanager_/api/v2/{projectId}/testcases

OAuth required scopes: TM.TestCases or TM.TestCases.Read

Required permissions: TestCase.Read

## EXAMPLES

### Example 1
```powershell
PS Orch1Tm:\> Get-TmTestCase
```

Retrieves all test cases from the current Test Manager project.

### Example 2
```powershell
PS Orch1Tm:\> Get-TmTestCase LoginTest
```

Retrieves the test case named "LoginTest" from the current project.

### Example 3
```powershell
PS Orch1Tm:\> Get-TmTestCase *API*
```

Retrieves all test cases whose names contain "API" using wildcard matching.

### Example 4
```powershell
PS Orch1Tm:\> Get-TmTestCase -Recurse
```

Retrieves all test cases from all Test Manager projects recursively.

### Example 5
```powershell
PS Orch1Tm:\> Get-TmTestCase -Path Orch1Tm:\MyProject
```

Retrieves all test cases from the specified Test Manager project without changing the current location.

### Example 6
```powershell
PS Orch1Tm:\> Get-TmTestCase -Recurse | Where-Object {$_.automationTestCaseName -ne $null}
```

Retrieves all automated test cases (those with associated automation) across all projects.

### Example 7
```powershell
PS Orch1Tm:\> Get-TmTestCase -Recurse | ConvertTo-Json -Depth 3
```

Retrieves all test cases and displays their complete structure in JSON format for detailed analysis.

### Example 8
```powershell
PS Orch1Tm:\> Get-TmTestCase -Recurse | Select-Object name, version, automationProjectName, updated | Format-Table
```

Displays test cases with key information in a formatted table showing name, version, associated automation project, and last update time.

### Example 9
```powershell
PS Orch1Tm:\> Get-TmTestCase -Recurse | Group-Object automationProjectName | Select-Object Name, Count
```

Groups test cases by their associated automation project and shows the count for each project.

## NOTES

This cmdlet operates on folder entities, which means you may need to navigate to specific Test Manager projects or use the -Path parameter to target the desired location.

For Test Manager operations, ensure you are connected to the appropriate Test Manager drive (Orch1Tm:, Orch2Tm:, etc.) or use the -Path parameter to specify the correct drive.

When using the -Recurse parameter with large Test Manager environments, the operation may take considerable time to complete. Consider using -Path to limit the scope if needed.

Test cases contain important information including:
- automationTestCaseName: The associated automation workflow file
- automationProjectName: The automation project containing the test
- version: The test case version for change tracking
- preCondition: Prerequisites for test execution
- inputParams: Parameters required for test execution

Use ConvertTo-Json to explore the complete structure of test case objects, as they contain detailed metadata that may not be visible in the default display format.

Parameter placement is important for folder entity cmdlets. Place -Path, -Recurse parameters immediately after the cmdlet name for proper auto-completion functionality.

Test cases can be either manual (no automation association) or automated (linked to UiPath workflows). Use filtering to distinguish between these types based on the presence of automationTestCaseName.
